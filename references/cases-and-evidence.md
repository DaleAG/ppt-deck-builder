# Cases and Evidence

## Core rule
Use cases before principles.

## Good case selection
Prefer these first:
- easy to understand
- low ambiguity
- easy to explain to non-experts
- clearly linked to the deck's core claim

## Case layering method
Layer 1: most stable, best for the deck body
- strongest proof
- easiest for bosses / teams to understand

Layer 2: interesting, but explain carefully
- good for imagination space
- useful for advanced possibilities

Layer 3: eye-catching but risky
- use for upper-bound or boundary discussion
- do not make it the default recommendation

## Rule for missing cases
If a user says key cases were lost:
- do not just paste the category list back
- merge those missing case dimensions into the actual page narratives

## Example page structures
### Two-column case page
Left: the case itself
Right: what it proves

### Three-card case page
Use when showing adjacent variations of one capability.

### Case progression
- simple / stable case
- more embedded workflow case
- advanced / upper-bound case
