# Story Architecture

## Core principle
A deck is not a document dump. It needs a spine.

## Useful deck patterns

### Pattern A: Four-question spine
Best for internal education / quick understanding decks.
- What is it?
- Why now?
- What value does it create?
- How should we act?

### Pattern B: Problem → Shift → Evidence → Recommendation
Best for strategy / proposal decks.
- what changed
- why current understanding is insufficient
- what evidence supports the new view
- what action is recommended

### Pattern C: Concept → Cases → Method → Boundary
Best for AI / product / emerging-tech decks.
- explain the object clearly
- prove it through cases
- abstract best practices
- explain limits and risk

## Title rules
- For boss-facing decks, prefer statement titles.
- Use question titles only when they create useful suspense.
- If a title sounds like chat, rewrite it.

## Mid-deck rules
- The most important pages are usually the middle proof pages.
- If those pages are weak, the whole deck feels empty even if the cover is strong.
- Cases should not be pasted as taxonomy; they should drive the page logic.

## Closing rules
A good ending should do one of these:
- summarize the recommendation
- explain the adoption path
- restate the takeaway with more confidence
